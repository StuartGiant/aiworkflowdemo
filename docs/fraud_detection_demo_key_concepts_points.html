Fraud Detection Demo — Key Concepts & Interview Talking Points

Round 2 Technical Interview — Xiaomi Singapore

1. The Pitch

This is an end-to-end, self-contained fraud detection system built on the IEEE-CIS Fraud Detection dataset (590,540 transactions, 3.50% fraud rate). It goes beyond a notebook model score — it demonstrates the full lifecycle a production fraud system needs: feature engineering across four signal layers, a calibrated probabilistic model, a cost-aware operating point tied to real analyst capacity, adversarial robustness testing, and per-transaction explainability for compliance and analyst trust.

The interactive HTML dashboard is the centerpiece: it lets an interviewer manipulate a transaction's attributes live and watch the fraud probability, decision tier, SHAP reason codes, and operational health metrics update in real time — including what happens when fraudsters adapt their behavior (adversarial drift) and when the system's monitoring would catch it (PSI alarms).

One-sentence pitch: "I built a fraud detection pipeline that doesn't just predict — it prices decisions by cost, explains itself per-transaction, and knows when it's being gamed."

2. Problem Framing & Business Context

  • Class imbalance is extreme: 3.5% fraud rate means a naive accuracy-optimizing model would achieve 96.5% accuracy by predicting "never fraud." This is why PR-AUC (precision-recall AUC), not ROC-AUC, is the primary metric — ROC-AUC is misleadingly optimistic under heavy imbalance because the false-positive rate denominator (all legitimate transactions) is huge.
  • Asymmetric costs: missing a fraud case (false negative) costs ~$420 (average order value + chargeback penalty + dispute overhead), while a false positive costs ~$6 (analyst review time + customer friction). That's a 70:1 cost ratio — the model must be tuned around this asymmetry, not around a 0.5 default threshold.
  • Operational constraint: a fraud team can only manually review a finite number of cases per day (assumed 200/day for this demo). The "mathematically optimal" threshold is often operationally unusable if it floods the queue.

3. Feature Engineering — What Was Built and Why

Features were organized into four signal layers, each targeting a distinct fraud pattern:

LayerFeaturesFraud pattern it detects
Velocityvelocity_count_1h/24h, velocity_value_1h/24h (rolling per card1)Card testing / rapid-fire fraud — many small or large transactions in a short window
Ratioc_total, c_max (sum/max of C1–C14 Vesta counters), r_email_missing, p_email_missingAccount history depth — thin/new accounts vs. established ones; missing email metadata correlates with disposable/throwaway identities
Interactionnew_acct_highval_express (new account + high value + express shipping), addr_age_days (from D1)Classic "hit and run" fraud: brand-new account, big purchase, fast shipping to avoid detection before chargeback lands
Networkemail_domain_mismatch, device_linkage_count, email_domain_linkage_countFraud rings — many transactions sharing one device or email domain indicates a coordinated attack, not organic customer behavior

Plus: has_identity flag (24.4% of transactions have an identity record — absence itself is predictive), and dist1 imputation (59.7% null — median-imputed with a dist1_missing binary flag so the model can distinguish "far from billing address" from "we don't know").

Criteria for choosing/dropping features

  • Keep if it encodes an independent fraud mechanism — velocity, ratio, interaction, and network layers each capture a different modus operandi, so they're complementary rather than redundant.
  • Drop raw high-cardinality identifiers that don't generalize: card2, card3, card5, addr1, addr2 (raw address/card codes) — these are near-unique per customer and would let the tree memorize training rows rather than learn patterns (overfitting / leakage risk).
  • Drop raw string columns already captured by an engineered feature: P_emaildomain/R_emaildomain are dropped in favor of email_domain_mismatch and email_domain_linkage_count; DeviceInfo/DeviceType are dropped in favor of device_linkage_count. This avoids double-counting the same signal in two forms and keeps the model interpretable.
  • Drop low-marginal-value categorical noise: M1–M9 match flags and several id_* string identity columns were dropped — they're either highly collinear with features already engineered or too sparse/noisy post-encoding to earn their complexity cost.
  • Always add a missingness flag when imputing (dist1_missing) — missingness is often more predictive than the imputed value itself, and blending them into one column would destroy that signal.

4. Model Selection — Why XGBoost

CandidateWhy not chosen (or when it would be)
Logistic RegressionFast and interpretable, but fraud signals are highly non-linear and interact (e.g., "new account AND high value AND express" is a 3-way interaction) — LR would need extensive manual feature crosses to match tree-based performance.
Random ForestHandles non-linearity and interactions well, but XGBoost's gradient boosting typically achieves better PR-AUC on tabular fraud data because each tree explicitly corrects the residual errors of the previous ensemble, rather than averaging independent bagged trees.
Deep Neural Net (MLP/TabNet)Needs much more data and careful regularization to beat gradient boosting on tabular data of this size (~590K rows); also harder to explain per-transaction without extra tooling (SHAP works natively and exactly on trees via TreeExplainer, only approximately on NNs).
XGBoost (chosen)Native handling of missing values, built-in class-imbalance handling via scale_pos_weight, excellent tabular performance, fast training via histogram-based splits, and exact SHAP explainability via TreeExplainer — which this project needs for adverse-action reason codes.

5. Hyperparameter Search — RandomizedSearch vs Optuna (TPE), Head-to-Head

The Section 4 configuration (max_depth=6, learning_rate=0.05, ...) was hand-picked. To demonstrate systematic tuning rather than guesswork, two search methods were run head-to-head on identical terms — same search space, same 3-fold CV, same 25-trial budget, same 60,000-row stratified subsample (kept small so ~150 total CV fits stay tractable; fixed at 200 trees during search, no early stopping) — then compared on both cost (wall-clock time, trials-to-95%-of-best) and performance (final CV PR-AUC, then full-scale held-out test metrics).

RandomizedSearchCVOptuna (TPE)
Best CV PR-AUC (search phase)0.52670.5464
Total search time (25 trials)140.8s155.9s
Trials to reach 95% of its own best312

Search-phase verdict: Optuna. It found a meaningfully better configuration (0.5464 vs 0.5267 CV PR-AUC) even though RandomizedSearch converged to "good enough" faster. Reasoning: each fit here is non-trivial (XGBoost on a mixed continuous/discrete 8-parameter space), and the trial budget is fixed and small — TPE's posterior-guided sampling exploits information from earlier trials that blind random sampling throws away, so it keeps finding better regions of the space later in the budget where random search has plateaued (visible in the convergence chart — Optuna's best-so-far line keeps stepping up past trial 12, RandomizedSearch's flatlines after trial 3).

Final full-scale evaluation — the twist

Both methods' best configs were then retrained at full scale (413K training rows, 500 estimators with early stopping, isotonic-calibrated) and scored on the same held-out test set used everywhere else in this demo:

ConfigPR-AUCF-β=2Est. cost/day*
Manual baseline (Section 4)0.64480.5295$29,797
RandomizedSearch-tuned0.76530.6726$21,221

*Cost/day here is illustrative: computed at a fixed threshold of 0.5 (not the capacity-constrained operating threshold from Section 6), so all three configs are compared on equal footing rather than each getting its own re-optimized threshold.

The interesting result: the ranking flips. Optuna won the CV search phase, but RandomizedSearch's config narrowly won the full-scale test evaluation (0.7653 vs 0.7618). fraud_xgb_tuned.pkl ships the RandomizedSearch-tuned model as a result. Both tuned configs crushed the manual baseline either way — PR-AUC +18.6–18.9%, cost/day down ~29–33%.

Talking point: this is a genuine, not manufactured, illustration of why you validate at production scale rather than trusting the cheap search proxy blindly. The 60K-row/200-tree/3-fold CV score is a fast approximation for ranking candidates during search, but it isn't the same objective as the final 413K-row/500-tree/early-stopped fit — variance in a small proxy can reorder close candidates. The practical lesson: use the fast proxy to prune a large space efficiently (that's still where Optuna's sample efficiency pays off against a much larger space or budget), but always confirm the final pick with a full-scale run before shipping — never promote straight from the search leaderboard.

6. Calibration — Why Isotonic Regression

Raw XGBoost output is a good ranking score but not a true probability — scores tend to be overconfident near 0 and 1. Isotonic regression is fit on a held-out 10% calibration split (never seen during training) to map raw scores to calibrated P(fraud), which matters because:

  • The cost-based threshold math (C_FN × FN + C_FP × FP) only makes sense if the probability output is a genuine estimate of fraud likelihood.
  • The response-tier UI (Allow / Step-up / Review / Block) presents probabilities directly to end users/analysts — an uncalibrated score would mislead them about true risk.

Isotonic (non-parametric, monotonic) was chosen over Platt scaling (sigmoid) because it makes no assumption about the shape of the miscalibration curve — appropriate given XGBoost's boosting can produce non-sigmoidal score distortions. Trade-off: isotonic needs more calibration data to avoid overfitting the step function, hence the dedicated 10% split rather than reusing the training set.

Implementation note: sklearn 1.2+ removed CalibratedClassifierCV(cv='prefit'), so calibration is done manually — fit IsotonicRegression directly on (raw_cal_scores, y_cal) and apply .predict() at inference time.

8. Adversarial Robustness — Planning for Fraudsters Who Adapt

Threat model simulated: "transaction splitting" — fraudsters learn that large transactions and velocity spikes get flagged, so they split purchases into micro-amounts (≤$50) spread across time to stay under both the amount and velocity signal thresholds.

Simulation method: the transformation is applied to all fraud rows plus a random 30% of legitimate rows (population-level shift, not just fraud-only) so the resulting distribution shift is realistically detectable by monitoring, not just a partial gaming of fraud examples.

MetricBaselinePost-drift
Recall @ operating threshold68.2%60.5% (−11.3% relative)
PR-AUC0.64570.5915
PSI — TransactionAmt~0.003 (stable)0.319 (alarm — threshold is 0.2)
PSI — velocity_value_1h~0.001 (stable)0.104 (below alarm, but rising)

Why Population Stability Index (PSI): PSI is the standard credit-risk/fraud industry metric for monitoring feature drift because it's interpretable (values below 0.1 = stable, 0.1–0.2 = moderate shift, >0.2 = significant shift requiring action) and works on any distribution shape. Quantile-based binning (rather than uniform bins) was used here because financial amount data is heavily right-skewed — uniform bins would concentrate 90% of transactions in one bucket and miss the shift entirely (this was an actual bug caught during development — initial uniform-bin PSI read 0.000 even after a real, detectable shift).

The operational lesson: when PSI crosses the alarm threshold, the demo surfaces a "Trigger retrain / champion-challenger" action — the point being that recall degradation from adversarial adaptation should trigger a monitoring alert and retraining workflow automatically, not wait for a quarterly model review.

9. Explainability — SHAP Reason Codes

  • Why SHAP: TreeExplainer computes exact Shapley values for tree ensembles in polynomial time (not the exponential brute-force cost of general SHAP), so per-transaction explanations are computationally practical at scale.
  • Why it matters for this business: adverse-action reason codes are often a regulatory/compliance requirement when declining or flagging a customer transaction — you must be able to state *why* a specific transaction was flagged, not just that the model said so.
  • What's shown: top-5 features by absolute SHAP value per transaction, color-coded red (increases fraud risk) / green (decreases risk), which doubles as an analyst triage aid — the analyst sees the same "why" the model saw before deciding whether to escalate.

11. Key Metrics Summary

  • Dataset: 590,540 transactions, 3.50% fraud rate, 24.4% identity join coverage
  • Model: XGBoost (500 trees, early-stopped) + isotonic calibration
  • PR-AUC: 0.6457 (manual baseline config)
  • Deployed threshold: 0.116 (capacity-constrained) → 47.6% precision, 67.0% recall, ~194 flags/day (within 200/day capacity)
  • Cost-optimal threshold: 0.013 → 92.2% recall but ~1,217 flags/day (6× over capacity — not deployable)
  • Adversarial stress test: transaction-splitting attack drops recall 11.3% relative and pushes PSI to 0.319 (above the 0.2 alarm threshold), confirming the monitoring layer would catch this drift
  • Hyperparameter search: RandomizedSearch vs Optuna (TPE), 25 trials each on identical CV/budget — Optuna won the search phase (CV PR-AUC 0.5464 vs 0.5267), but RandomizedSearch's config won at full scale on held-out test (PR-AUC 0.7653 vs 0.7618) — +18.6% PR-AUC over the manual baseline, shipped as fraud_xgb_tuned.pkl

12. Anticipated Interview Questions

  • "Why not just use accuracy?" — With 3.5% fraud, a model that predicts "never fraud" scores 96.5% accuracy while catching zero fraud. PR-AUC and recall-at-capacity are the metrics that actually matter here.
  • "Why isotonic over Platt scaling?" — No assumption on the shape of the miscalibration function; XGBoost's boosted score distortions aren't guaranteed to be sigmoidal.
  • "How would you retrain this in production?" — Tied to the label-lag banner: wait for chargeback ground truth to mature (60–90 days), then use the PSI alarm as the trigger for a champion/challenger retrain rather than a fixed calendar schedule.
  • "Why compare two search methods instead of just picking one?" — Because "which search method" is itself a decision worth justifying with evidence, not a default. Running both on identical terms shows the actual cost/performance trade-off rather than asserting Optuna is better from first principles — and the result (search-phase winner ≠ full-scale winner) is a genuinely useful finding, not just a demo flourish.
  • "Why did the winner flip between the search phase and the final evaluation?" — The search phase scores a 60K-row/200-tree/3-fold CV proxy for speed; the final evaluation is a 413K-row/500-tree/early-stopped fit — a different, more expensive objective. Close candidates on a fast proxy can reorder under the real objective. Lesson: use the proxy to prune a large space cheaply, but always confirm the final pick with a full-scale run before shipping.
  • "Would Bayesian search always beat random search?" — Not necessarily — TPE's advantage grows with a larger search space or trial budget where exploiting the posterior compounds; with very few trials or a cheap/embarrassingly parallel budget, random search's simplicity and even coverage can be competitive. Here, with a fixed 25-trial budget and non-trivial per-trial cost, TPE's sample efficiency paid off in the search phase, though it wasn't decisive at final scale.
  • "What would you do differently with more time?" — Graph-based network features (device/email clustering via community detection rather than simple linkage counts), a proper backtesting harness simulating label lag over time, a champion/challenger deployment pipeline instead of the single static model shown here, and running the hyperparameter search directly on the full-scale training set (or a larger subsample) to remove the proxy/final-objective mismatch entirely.
